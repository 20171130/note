## 2026-05-25 21:50 PDT — 22:31 PDT: CTT tokenizer S1+S2 — data + tokenizer + condition orderings

Built the input pipeline for the [Continuous Token Transformer](/CTT/README.md). Two modules:

1. **`CTT/data.py`** — OMol25 LMDB → `Record` dataclass. All 30 metadata fields surfaced (identification, DFT setup, system counts, per-atom, scalars, diagnostics). Reader is `lmdb.open` + `zlib.decompress` + `json.loads` + ASE `__ndarray__` envelope unpack — same path as `omol25_eval/eval_baseline.py:44`.
2. **`CTT/tokenizer.py`** — `CTTTokenizer` wraps any HF tokenizer; output is a heterogeneous `list[int | float | np.ndarray(3,)]`. JSON-literal framing through the HF text vocab; vec3 / scalar pass through as raw values (no decimal-text round-trip). Qwen2.5-0.5B as default backend; reachable via `CTTTokenizer.from_pretrained("Qwen/Qwen2.5-0.5B-Instruct")`. Round-trip exact on real OMol25 records; 1548 tokens for a 112-atom elytes record (~14 tokens/atom).

### Design decisions Henry pushed for (vs my first proposals)

- **JSON-literal framing**, no reserved structural specials for `{}`/`[]`/`,`. Versatility argument: any nested dict serializes, arrays terminate naturally with `]`, keys are plain strings. My initial proposal was schema-driven; Henry chose JSON-like for maximal generality. Solved field-presence / length-prefix / key-encoding questions in one stroke.
- **Heterogeneous token stream** (`int | float | np.ndarray`), not parallel placeholder + sidecar arrays. The downstream embedder dispatches on type. (Sidecar form remains an internal batching trick.)
- **Small-int reservation.** Bare integers like `6` should be one token; the string `"6"` stays three tokens (`"`, `6`, `"`). I cache `tok.encode(str(i))` at construction for `0..1023`; on Qwen2.5 only 0..9 are single-token (Llama 3 would give 0..999) — flagged as a swap point if we change backend.
- **Single source of truth for field order = Record dataclass declaration.** Initially I had three parallel constants (`OMOL25_PROMPT_FIELDS` / `_PREDICT_ORDER` / `_DEFAULT_ORDER`). Henry: "can we define the order using Record fields? not a separate list." Reduced to: `to_dict()` walks `dataclasses.fields(self)` and skips `raw` by name. To reorder, edit the dataclass.
- **Maximal metadata.** I had a 6-field Record; Henry: "include as much metadata as possible". Expanded to 30 fields. Pretraining-time information richness > inference-time cleanliness; "diagnostic" fields (n_scf_steps, core_hours, warnings) get to predict-tail, not dropped.
- **Condition-set + builder, not per-task tuples.** I drafted two `OMOL25_GEN_*` tuples for size→chemistry and spec→structure. Henry: "let's not list them one by one … support specifying ordering via condition fields." Replaced with `ordering_from_conditions(conds, domain=...)` that produces `[domain] + [conditions] + [targets]` all in declaration order. Presets `OMOL25_COND_{NONE, SIZE, COMPOSITION, CHEMISTRY, STRUCTURE}` cover the size⊂composition⊂chemistry⊂structure hierarchy; inverse design is a custom tuple.

### Two ordering families ([CTT/README.md#field-ordering-for-omol25](/CTT/README.md))

- **Predictive** (default = dataclass order): structure first, derived counts / observables / diagnostics fall out. Models the MLIP forward pass + property regression. Key invariant: derived fields (composition, num_atoms, num_electrons, n_basis, fmax) come after the fields they're derived from, so the prediction is a trivial sum/count/max.
- **Generative** (condition-set parameterized): the typical condition combinations are recorded as presets; the full sequence is `[domain] + [conditions] + [targets]`. Removes the combinatorial explosion of per-task tuples.

### Findings worth keeping

- **Qwen2.5 single-token integer range is 0..9 only.** Verified via probe. Limits per-atom-id compactness vs Llama 3 (0..999). Decision-relevant when picking the base LLM.
- **OMol25 stream-token breakdown** for a 112-atom elytes record: 1089 text-vocab ids (framing + warnings + key names + small ints), 235 floats (per-atom partial charges + electronic-property scalars), 224 vec3 (positions + forces), 1 final scalar (energy). ~14 tokens/atom dominated by per-atom data; framing overhead is constant per record.
- **`integrated_densities` is `[Nα, Nβ, total]`, NOT a spatial vec3.** My tokenizer auto-treats any float[3] ndarray as vec3, so `_to_record` converts this field to a Python list before handing off. Edge case to remember if we add other 3-scalar fields downstream.
- **The `cell` and `pbc` top-level fields are present but meaningless** for OMol25 (molecules only, `pbc` always `[False, False, False]`). Dropped from `Record`; would matter for OMat24 / OC20.

Three follow-ups landed in this session.

### `data.py`: atom-shuffle augmentation
Added `Record._PER_ATOM_FIELDS = (numbers, positions, mulliken_charges, lowdin_charges, forces)` and `Record.shuffle(rng)` — one permutation applied across all per-atom arrays so atom `i` stays consistent across fields. `iter_shard(..., shuffle_atoms, rng)` and `OMol25Dataset(..., shuffle_atoms, seed)` expose it. ### `tokenizer.py`: respect-the-base-tokenizer principle
Henry's call: "make the text tokenizer compatible with whatever tokenizer we init from pretrained, so do not reserve int tokens in our own way, respect original." Removed `_int_one_tok` / `_tok_minus` / `_emit_int` / `small_int_cache` ctor arg. Integers now go through `hf.encode(str(n))` directly — whatever the base does is what we emit. Stream length unchanged on Qwen2.5 OMol25 records (Qwen already merges short int strings well enough).

Documented as the load-bearing principle for the whole tokenizer in both `README.md#input` (new lead bullet "Compatible with the base text tokenizer — intentional, for pretrained init") and the `tokenizer.py` module docstring. Framed as: our text vocab is exactly a subset of the base's so embedding matrix + LM head load mechanically. Integer handling is a *direct consequence*, not a separate design choice.

### Plan: embedder ↔ decoder pair ([`/.llms/plans/ctt_continuous_embedder_and_decoder.plan.md`](/.llms/plans/ctt_continuous_embedder_and_decoder.plan.md))

Drafted, then iterated through six rounds of Henry pushback:

1. **Field tag → field-agnostic.** My v1 had per-field box `L_f` and a routing tag from the JSON context. Henry: "field tag should be removed for versatility." Dropped to one shared spec.
2. **Range-free entirely.** My v2 still had a global box `L_vec` + signed-log scale `(s, L_scl)`. Henry: "I completely removed the dependence of range" — pointing me back at the README's IEEE-bit decomposition in `## In-token AR head`. Sign + exponent + mantissa MSB→LSB. Exponent bits *are* log-magnitude, so auto-calibrates from data; no probe, no box, no normalization constant. For vec3: same K, per-axis IEEE bits joined into a `V_d = 8` joint subtoken per depth.
3. **No special-pattern canonicalization.** I had NaN/±Inf/subnormals canonicalized at encode. Henry: "we can encode and decode any binary." Dropped — codec is bit-exact on the full `2^K` pattern space.
4. **No weight tying.** I had E_d ↔ W_d tied (text-LM trick analogy). Henry: "should we do weight tying at all? we may do the opposite … we do not have translational equivariance for bits." Independent params per role; left tying behind an ablation flag.
5. **One-letter `E_d/F_d/W_d` → role names.** Henry: "what is E_d F_d w_d?" Renamed to `enc_emb`, `dec_emb`, `dec_head` with plain-English roles in the glossary.
6. **`Encoder` and `Decoder` are separate classes, not an `InnerCodec` wrapper.** Henry: "encoder and decoder are different classes, do not mix them up when discussing." Also "shape should be V_d x D x K, since they work both as input and positional embedding here" — folded per-depth K-positional and V_d-content into one tensor per role.

Also renamed `ieee_bits_*` → `scalar2bits` / `vector2bits` and moved them to `utils.py` (not a new `discretize.py` — "we are not discretizing, they are discrete approx of continuous from the beginning").

### Findings worth keeping

- **IEEE bits *are* the calibration.** Removing the box parameter wasn't just simplification — it's principled. The exponent bits give the model log-magnitude directly, so the entire dynamic-range concern dissolves. The cost is the model has to learn the exp/mant asymmetry from data; benefit is zero per-dataset / per-field bookkeeping.
- **`V_d × D × K` table shape**. Each `(bit value, depth)` cell is its own token type — bit=1 at depth 0 (sign) is genuinely a different object from bit=1 at depth 8 (exp LSB). Folding both axes into one learnable tensor is the natural data structure; separate position-encoding addition is unnecessary.
- **Two distinct token types added to base vocab**: `<scalar_token>` and `<vector_token>`. These are the *only* vocab additions on top of the base tokenizer — preserves the "text vocab = subset of base" property we settled on this session.

### Henry's learning signals
- **Re-direct to existing artifacts.** When I drifted from the design recorded in `README.md#in-token-ar-head`, Henry quoted it back at me verbatim. Lesson: before designing the next layer, re-read the design notes in the doc I'm extending, not just the immediately-preceding chat.
- **Decisive minimalism over knobs.** Henry consistently removed configurability I added (field tags, range constants, canonicalization, tying). Net direction: fewer hyperparameters, fewer code paths, fewer side-channels. The codec ended up much simpler than v1.
- **Naming carries meaning.** "Discretize" was wrong because floats are already discrete; "vec3" leaked an implementation detail into the API; one-letter `E_d` was too dense to parse. Each rename was a clarity win.
