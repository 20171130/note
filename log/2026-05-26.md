## 2026-05-26 01:06 PDT — Codec is the real bottleneck; h-broadcast fixes it; capacity floor d_inner=128

Scope: Henry called out that "100% on one OMol25 record" was insufficient evidence — that test only proves the outer LLM can memorize 459 specific floats through the codec, not that the codec itself is invertible on the float distribution. "We need 100% acc for all float vectors given enough data, or the encoder/decoder will become bottleneck eventually." Built the proper test, found the codec WAS the bottleneck, fixed it.

### Pure codec autoencoder test ([`tests/codec_saturation_test.py`](/CTT/tests/codec_saturation_test.py))

Removes the outer LLM entirely: `Encoder(x) → h → Decoder(h) → x̂`, train on random float32s drawn from log-uniform exponents [-30, 30] × {±1 sign} × mantissa [1, 2), evaluate bit-exact on held-out random samples. If the codec fails here, no amount of outer-LLM scaling can save it.

### First run was a disaster at the same capacity that aced the one-record smoke

`d_model=256 d_inner=256 n_depth_layers=2 tied 6000 steps batch 512`:
- **scalar: 380/2048 = 18.6% bit-exact** (final per-depth loss 0.06)
- **vector vec3: 1642/2048 = 80.2% bit-exact** (final per-depth loss 0.02)

Per-depth loss 0.06 ⇒ ~95% per-bit acc ⇒ 0.95³² ≈ 19% reconstruction — matches scalar exactly. Vector tolerates this better because its per-depth alphabet is V=8 vs V=2 so per-step loss is cleaner.

### Root cause — decoder's `h` had to travel K-1 causal hops

The decoder's depth recurrence built x[0]=in_proj(h) and x[d≥1]=dec_emb[bit_{d-1}, :, d-1]; only position 0 saw `h`. By depth 31 the conditioning signal had to be retrieved through 30 hops of causal self-attention against feedback embeddings that carry no `h` of their own. Information bleeds out, codec fails.

### Fix: broadcast `in_proj(h)` additively into every depth position

`inner_codec.py:Decoder._depth_states` and `._sample_bits`: `x[d] = in_proj(h) + dec_emb[bit_{d-1}, :, d-1]` for d≥1, plus `x[0] = in_proj(h)`. Same training config, same params:
- **scalar: 2048/2048 (100%)** train loss 0.00000
- **vector vec3: 2048/2048 (100%)** train loss 0.00001

(a) was the right diagnosis, b/c (encoder pool, capacity bumps) weren't needed.

### Capacity floor sweep ([`tests/codec_capacity_sweep.py`](/CTT/tests/codec_capacity_sweep.py))

At fixed `d_model=256, n_depth_layers=2, n_heads=2, tied, 3000 steps batch 512`:

```
d_inner  params   scalar bit-exact      vector vec3 bit-exact
   128   1.17M   1024/1024 (100.0%)    1024/1024 (100.0%)
    64   0.32M   1023/1024 ( 99.9%)    1022/1024 ( 99.8%)
    32   0.10M    526/1024 ( 51.4%)       1/1024 (  0.1%)
    16   0.03M     29/1024 (  2.8%)       0/1024
     8   0.01M      0/1024                0/1024
```

**99.9% is not 100% — `d_inner=128` is the true lower bound.** `d_inner=256` is the safe default. The cliff between 64 and 32 is sharp (3× param reduction kills it); vector's cliff is harsher than scalar's because V=8 makes per-depth errors more costly.

### Henry's learning signals
- **Saturation evidence must come from a generalization test, not a memorization test.** "100% bit-exact on one record" is a corruption of the bit-exactness claim if the model has enough capacity to memorize that record's bytes. The minimum sound test is autoencoder on diverse held-out samples. Adopt as a default: when claiming an architecture saturates, the saturating test must put the claim at risk on held-out data.
- **Don't accept 99.9% as 100%.** Henry's "we need 100%" enforcement caught me about to land `d_inner=64` as the floor on the strength of 1023/1024. Floor of a saturating curve is the smallest config that hits *exactly* the target on the eval set, not "close enough at this many steps". Treat any non-100% as a fail for purposes of identifying a capacity floor.
- **Architectural fixes can be much cheaper than capacity bumps.** Before doing the sweep I had three hypotheses ranked (a) decoder h-injection / (b) encoder pool / (c) bump capacity. Henry chose (a) ("natural, try it first"). (a) alone moved scalar from 18.6% → 100% at the same width. Lesson: when the failure mode has a specific information-flow story, prefer the targeted fix to brute capacity scaling.

## 2026-05-26 03:14 PDT — Amendments to 03:05 review (after Henry pushback)

Two findings revised.

### B2 — diagnosis flipped: bug is in training, not inference

Henry's design intent (clarified in chat): both training and inference predict whatever sits at position i+1 (text id OR continuous value) from `h[i]`. The special token's own hidden is never the codec's input — it exists at position i+1 only so downstream attention sees a real value via the encoder-override.

That means INFERENCE (`evaluation.py:166-171`) is already correct: `h[:, -1, :]` is `h[i]` at the moment the LM head predicts a special id at i+1; passing it to `decoder.sample_*` is the right conditioning. TRAINING (`model.py:267-274`) is the side that needs to change: `h_at = h[b_idx, p_idx]` should be `h[b_idx, p_idx - 1]` so the codec loss is computed at the prior hidden, matching how the LM head's CE is computed.

Consequences of the fix:
- Encoder override at the slot position stays — only its role narrows to "give downstream tokens the real value to attend to", not "be the codec's input".
- Codec is no longer an autoencoder during training; it becomes a true prediction head conditioned on the LM's context. The `codec_saturation_test` 100% guarantee was an autoencoder probe through a single hidden; it bounds the codec's representational ceiling, not what the outer LM can actually deliver to it. Real-world bit-exact rates will live somewhere between 0 and that ceiling.
- The 1-record overfit smoke will still PASS — constant value from constant context is a degenerate prediction problem, memorizable in the outer LM's residual.

Original 03:05 framing ("inference uses wrong hidden, training is right") came from reading the saturation-test conditioning back into training; that's how the autoencoder was probed but not how the architecture is meant to be trained for AR decode. Corrected in place because the wrong diagnosis would have steered Henry to fix the wrong file.

### B5 — autocast scope clarification

Earlier framing implied `.float()` on a module "doesn't beat autocast" without saying what autocast actually touches. Precise:
- `torch.autocast` only intercepts a fixed registry of ops (mostly `nn.Linear`, matmul, conv, SDPA). It rewrites their COMPUTE dtype and OUTPUT dtype to the autocast dtype (bf16 in our case).
- It does NOT change: parameter tensor dtype, raw input tensors held in memory, gathers / indexing, integer ops, custom Python tensor manipulation.
- Therefore your IEEE float32 record value passes through `scalar2bits` (→ long bit indices) and `enc_emb[bits]` (gather → fp32 lookup) unaffected.
- But the next step — the encoder/decoder depth `TransformerEncoderLayer` plus `in_proj` / `out_proj` — runs Linear and matmul, so under bf16 autocast each layer's OUTPUT becomes bf16 regardless of weight or input dtype. Bit-prediction logits would end up in bf16; bf16's 7-bit mantissa is borderline for the saturation-test confidence margins.
- `.float()` on the module pins WEIGHT dtype only. The two knobs (parameter dtype, compute dtype) are independent.
- Fix is `with torch.autocast(device_type="cuda", enabled=False):` wrapping the codec forward. Combined with the existing `.float()` pin on parameters and the `h_at = ....float()` cast on input, the entire codec then runs in fp32 cleanly while Qwen3 stays in bf16 outside the context.

Updated [coding_caveats.md#Numerics and Errors](../knowledge/misc/coding_caveats.md) entry to be precise about the parameter-vs-compute distinction.

### D5 evaluator orchestrator (TF-only, commit `c932fd8`)
`evaluation.build_teacher_forced_evaluator(tokenizer, splits, subsample_frac, ordering)` returns the trainer's expected `(model, step) -> dict` callable. Per call:
1. RNG seeded by `step` picks a deterministic subsample per split.
2. Rank-shards the subsample (`sample[rank::ws]`) so each rank does its share.
3. `teacher_forced_eval(...)` per split → cross-rank `.reduce()` baked into `teacher_forced_eval` so rank-0's dict carries global aggregates.
4. Returns `eval/{split_stem}/loss/{text, cont/<field>, total}` plus `eval/{split}/n_records`.

AR + physical-unit MAE intentionally omitted: needs an AR-output parser (walk the token stream + numeric sidecar back to per-field `{name: value}` dicts feeding `compute_physical_mae`).
