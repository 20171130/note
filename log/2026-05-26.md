### Three mid-execution refinements Henry pushed

1. **`d_inner` decoupled from `d_model`** ("hidden dim should be a stand alone hyperparam, much smaller than LLM dim >1024, we can decrease it until overfit autoencoding fails"). Added `d_inner` ctor arg to both classes; tables and depth-transformer run at `d_inner`; `Encoder.out_proj = Linear(d_inner, d_model)` and `Decoder.in_proj = Linear(d_model, d_inner)` (Identity when equal). Methodology baked into the sweep script.
2. **Weight tying re-enabled and made default** ("now I think it might make sense to tie weights, let's test weights tied as well"). Earlier plan rejected tying ("no translational equivariance for bits"); the data flipped the decision. Implementation: pass `tied_encoder=enc` to `Decoder`; the decoder owns no bit-tables and resolves `dec_emb := enc.{kind}_emb`, `dec_head := enc.{kind}_emb.permute(1, 0, 2)`. Single `(V, d_inner, K)` tensor plays input / feedback / transposed-output roles. Default flipped to `tie_weights=True` in `ModelStub`. Sweep evidence: at `d_inner=128` tied uses ~7% fewer codec params *and* hits 99.6% vs 98.9% untied; tying is regularization-friendly at the small-codec scale.
3. **Saturation pushed to 100%** ("must achieve 100%"). Bumped smoke to `d_model=256 d_inner=256 n_outer_layers=2 n_depth_layers=2 tied Adam-3e-3 cosine 4000 steps`. Result: AR loss 2.94 → 3e-4, **459/459 bit-exact** greedy reconstruction. ~3 min on one H100.

## 2026-05-26 01:06 PDT — Codec is the real bottleneck; h-broadcast fixes it; capacity floor d_inner=128

Scope: Henry called out that "100% on one OMol25 record" was insufficient evidence — that test only proves the outer LLM can memorize 459 specific floats through the codec, not that the codec itself is invertible on the float distribution. "We need 100% acc for all float vectors given enough data, or the encoder/decoder will become bottleneck eventually." Built the proper test, found the codec WAS the bottleneck, fixed it.

### Pure codec autoencoder test ([`tests/codec_saturation_test.py`](/CTT/tests/codec_saturation_test.py))

Removes the outer LLM entirely: `Encoder(x) → h → Decoder(h) → x̂`, train on random float32s drawn from log-uniform exponents [-30, 30] × {±1 sign} × mantissa [1, 2), evaluate bit-exact on held-out random samples. If the codec fails here, no amount of outer-LLM scaling can save it.

### First run was a disaster at the same capacity that aced the one-record smoke

`d_model=256 d_inner=256 n_depth_layers=2 tied 6000 steps batch 512`:
- **scalar: 380/2048 = 18.6% bit-exact** (final per-depth loss 0.06)
- **vector vec3: 1642/2048 = 80.2% bit-exact** (final per-depth loss 0.02)

Per-depth loss 0.06 ⇒ ~95% per-bit acc ⇒ 0.95³² ≈ 19% reconstruction — matches scalar exactly. Vector tolerates this better because its per-depth alphabet is V=8 vs V=2 so per-step loss is cleaner.

### Root cause — decoder's `h` had to travel K-1 causal hops

The decoder's depth recurrence built x[0]=in_proj(h) and x[d≥1]=dec_emb[bit_{d-1}, :, d-1]; only position 0 saw `h`. By depth 31 the conditioning signal had to be retrieved through 30 hops of causal self-attention against feedback embeddings that carry no `h` of their own. Information bleeds out, codec fails.

### Fix: broadcast `in_proj(h)` additively into every depth position

`inner_codec.py:Decoder._depth_states` and `._sample_bits`: `x[d] = in_proj(h) + dec_emb[bit_{d-1}, :, d-1]` for d≥1, plus `x[0] = in_proj(h)`. Same training config, same params:
- **scalar: 2048/2048 (100%)** train loss 0.00000
- **vector vec3: 2048/2048 (100%)** train loss 0.00001

(a) was the right diagnosis, b/c (encoder pool, capacity bumps) weren't needed.

### Capacity floor sweep ([`tests/codec_capacity_sweep.py`](/CTT/tests/codec_capacity_sweep.py))

At fixed `d_model=256, n_depth_layers=2, n_heads=2, tied, 3000 steps batch 512`:

```
d_inner  params   scalar bit-exact      vector vec3 bit-exact
   128   1.17M   1024/1024 (100.0%)    1024/1024 (100.0%)
    64   0.32M   1023/1024 ( 99.9%)    1022/1024 ( 99.8%)
    32   0.10M    526/1024 ( 51.4%)       1/1024 (  0.1%)
    16   0.03M     29/1024 (  2.8%)       0/1024
     8   0.01M      0/1024                0/1024
```

**99.9% is not 100% — `d_inner=128` is the true lower bound.** `d_inner=256` is the safe default. The cliff between 64 and 32 is sharp (3× param reduction kills it); vector's cliff is harsher than scalar's because V=8 makes per-depth errors more costly.

### Henry's learning signals
- **Saturation evidence must come from a generalization test, not a memorization test.** "100% bit-exact on one record" is a corruption of the bit-exactness claim if the model has enough capacity to memorize that record's bytes. The minimum sound test is autoencoder on diverse held-out samples. Adopt as a default: when claiming an architecture saturates, the saturating test must put the claim at risk on held-out data.
- **Don't accept 99.9% as 100%.** Henry's "we need 100%" enforcement caught me about to land `d_inner=64` as the floor on the strength of 1023/1024. Floor of a saturating curve is the smallest config that hits *exactly* the target on the eval set, not "close enough at this many steps". Treat any non-100% as a fail for purposes of identifying a capacity floor.
- **Architectural fixes can be much cheaper than capacity bumps.** Before doing the sweep I had three hypotheses ranked (a) decoder h-injection / (b) encoder pool / (c) bump capacity. Henry chose (a) ("natural, try it first"). (a) alone moved scalar from 18.6% → 100% at the same width. Lesson: when the failure mode has a specific information-flow story, prefer the targeted fix to brute capacity scaling.

### Code landed
- `inner_codec.py` — `Decoder._depth_states` and `Decoder._sample_bits`: broadcast `in_proj(h)` to every depth input position.
- `tests/codec_saturation_test.py` — pure codec autoencoder on random floats; asserts 2048/2048 scalar and vector.
- `tests/codec_capacity_sweep.py` — d_inner ∈ {128, 64, 32, 16, 8} sweep, summary table.
- `README.md` Inner-codec section — rewrote "Inner vs outer width" + new "Capacity floor (pure codec autoencoder, not LLM memorization)" subsection with the sweep table; deleted the misleading "d_inner≈64 capacity floor for one record" claim.

- **WandB**: self-hosted at `meta.wandb.io`. Both `WANDB_API_KEY` and `WANDB_BASE_URL` are now in `~/.bashrc`. Key updated to the new one Henry shared today (replaced the older key in place).
- **Two long runs side-by-side, identical LRs for controlled comparison**: `init=pretrained` and `init=scratch`, same dataloader seed, both with `outer_lr=2e-5, codec_lr=3e-4, AdamW(WD=0.01, betas=(0.9, 0.95))`, cosine + 5% warmup. The init bit is the only thing that differs — if we used different LRs we couldn't tell whether a gap is from pretrained weights or from a hotter learning rate.
- **Phase 1 codec-only warmup**: outer transformer frozen, runs until codec eval CE plateaus (patience=3 evals, min_delta=1e-3, safety cap 20k steps). Henry's preferred adaptive stopping over a fixed step count.
- **Sequence packing** (not pad-to-longest) — modern LLM training norm. Pack records back-to-back into fixed-length rows with EOS separators; `cu_seqlens` for block-diagonal attention via FlashAttention-2's varlen path. `ctx_len` and `batch_per_gpu` chosen via a pre-flight `scripts/probe_throughput.py` sweep over `{2048, 4096, 8192, 16384} × {1, 2, 4, 8}` on 8×H100.
- **Per-field CE attribution**: every token (text AND continuous) gets attributed to its enclosing Record field via a `field_stack` walk in the tokenizer. Loss dict is token-averaged per field, with separate `text`, `cont`, and aggregate `total` series. Outer `{` / final `}` go under `__framing__`.
- **Continuous-token CE on the same scale as text-token CE**: reuse existing `Decoder.train_loss_*` and multiply by `K_s`/`K_v` to convert mean-over-B*K to per-token sum-over-K. No new method needed — earlier draft had over-engineered this; Henry's "is there anything nontrivial here? i don't get it" caught it.
- **Text-token CE on `<scalar_token>` / `<vector_token>` positions is NOT masked**: the outer LM is supervised to predict the special-id itself so it learns *when* to emit a continuous slot. Codec runs at the special-id's own hidden for the value. Earlier draft had wrongly masked these — Henry caught it.
- **FieldCEAccumulator does `.detach()` per entry** to avoid retaining the autograd graph for logging; trainer holds exactly one un-detached tensor (`total_loss`) for `.backward()`. Cross-rank aggregation under FSDP uses `torch.distributed.all_reduce(SUM)` of `(sum_ce, n_tokens)`; wandb is a sink not an aggregator. Earlier draft framed this as "thread-safe" — wrong framing; Henry's "does wandb offer similar things?" caught it.
- **Eval**: teacher-forced CE on all val_* + metal_ligand_ood at 10% subsample (~3 min); double-AR decode on 200 records/split (~3 min). Total ~5-6 min/eval, ~8 hr / 10% of 3-day budget at `eval_every=2500`. Both metrics emitted because they measure different things: teacher-forced isolates per-step quality, AR-decode shows compounding error (late fields see all upstream model mistakes). The OMol25 paper baselines (cited in `omol25_eval/eval_baseline.py:186-189`) are single-shot regressors, not AR — apples-to-apples requires reporting both AR-decode MAE and teacher-forced-prefix single-field MAE; the gap quantifies the compounding cost.
