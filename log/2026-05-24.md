## OMol25 on-disk inspection (2026-05-24 19:00–19:44 PDT)
OMol25 250430-release is on `/checkpoint/ocp/shared/omol/`. Probed `stats/` (pre-computed energy/force histograms + lin-ref) and `*.aselmdb` shards directly — full split table in [work/multimodal.md](/work/multimodal.md#omol25-on-disk).

Format: each `.aselmdb` is plain LMDB with integer string keys (`b'1'`, `b'2'`, …); value is **zlib-compressed JSON** of an ASE-dict-like record. The interesting metadata lives under `obj["data"]`: `data_id`, `charge`, `spin`, `num_atoms`, `composition`, etc. No fairchem install needed — `lmdb`+`zlib`+`json` is enough.

Decisive findings:
- `simple_*` = paper's "Neutral" subset (charge=0, spin=1 only; sources ANI2x/OrbNet/GEOM/Trans1x/SPICE/RGD).
- `uhs_ood` = **ultra high spin** OOD (spin multiplicity 12–21, |charge| up to 8), not "unique high spin".
- `metal_ligand_ood` count matches paper M-Lig; `elytes_ood` is all electrolyte; `train_4M` covers all 10 sources uniformly.
- On-disk `train` is ~105 M structures by per-shard ×n estimate vs the paper's 140 M "All" — discrepancy not yet investigated.

Procedural lesson: my first scanner did `list(txn.cursor().iternext(keys=True))` per shard on NFS — 429k keys × multiple shards × multiple splits → multi-minute hang and a 600s timeout. Fast scheme: open one shard, read `env.stat()["entries"]`, sample first 50 records sequentially with `cur.first()` + `cur.next()`. Total runtime <1s for all 9 splits. **Rule of thumb: on FSx, never enumerate all LMDB keys when you only need a sample.**

Also: system `python3` is bare; `python3 -m pip install --user lmdb ase numpy` works without sudo and persists to `~/.local`. Worth setting up a real env (uv/conda) before the actual model dev.

Henry — learning signal: (1) check shared-storage paths before downloading; (2) when probing big files on NFS, always design the access pattern to avoid full enumeration; (3) the gated HF DATASET.md is unnecessary — the data files themselves carry full per-frame provenance.
